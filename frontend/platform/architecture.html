Data flow · developer submission lifecycle

  1. Upload. Developer pushes a binary (or source) to submission svc over HTTPS. The service hashes (SHA-256), validates the language contract, and writes to object storage.
  2. Orchestrate. orchestrator reads the build manifest, builds an OCI image, and asks the runner to schedule a sandbox with the team's CPU/memory budget.
  3. Isolate. sandbox runner spawns the container under cgroups + seccomp + a dedicated network namespace; only the gateway can reach it.
  4. Load. load gateway opens FIX / WS / REST sessions on behalf of the bot fleet and forwards orders to the sandbox at line rate.
  5. Measure. Every request/response pair is tagged with wall-clock timestamps and streamed to Kafka. scoring consumes the stream, computes p50/p90/p99 + TPS + correctness.
  6. Rank. Scores land in Redis (sorted set). The leaderboard ws service broadcasts deltas to the public frontend (this leaderboard page).

Tech choices

Defensible decisions.

// transport · bot → sandbox

FIX 4.4 · binary framing

FIX is the institutional standard. Binary framing keeps headers under 20 bytes. We expose REST + WS for developers who want a simpler surface.

// metrics bus

Redpanda

Drop-in for Kafka; single binary; no JVM; ~10× lower p99 publish latency. Telemetry partitions by team_id so consumers parallel-process by team.

// storage

TimescaleDB + Redis

Timescale for chunked time-series telemetry (queryable forever). Redis sorted sets for the live leaderboard - O(log N) inserts, O(N) reads.

// isolation

Firecracker microVMs

~125ms startup. Hardware-level isolation. Strict CPU pin + RLIMIT + seccomp default-deny. No filesystem write outside /tmp.

// orchestration

K8s + KEDA

Scale bot fleet by Kafka lag. Submission jobs are K8s Jobs with TTL. Sandboxes are Pods with strict Resource Quotas.

// IaC

Terraform + Helm + ArgoCD

Terraform provisions cloud. Helm packages each service. ArgoCD reconciles git → cluster. Spin up the whole platform in one command.
