02 / Architecture

Four subsystems.
One contract.

Your solution must encompass these components. Decoupled, concurrent, resilient. Communication protocols, data stores, and isolation strategies are yours to choose and yours to defend.

// 01 · ingestion dockercgroupsseccomp

Submission & Sandboxing Engine

A secure pipeline where users upload binaries or source (C++, Rust, Go). Containerize and deploy submissions in strictly isolated environments. Malicious code stays caged; all users receive identical, fair hardware budgets.

  • CPU pinning & strict memory caps per submission
  • Filesystem + network namespace isolation
  • Reproducible build pipeline, signed artefacts
  • Hot-swap deploy with zero-downtime rollover
// 02 · load fixwsrest

Distributed Load Generator

The engine. A scalable traffic-generation service capable of spawning thousands of distributed bots simulating diverse market participants (high-velocity Limit Orders, Market Orders, Cancels) directed at the user's endpoints.

  • Horizontally scalable bot orchestration
  • Pluggable strategies: makers, takers, sniffers, cancels
  • Deterministic seeds for reproducible runs
  • Wire-level fidelity for FIX / gRPC / WebSocket

  • Correctness: price-time priority & fill validation
  • Streamed to the metrics bus, durable on disk

07 / FAQ

Common questions.

Everything you need to know about QuanTime's architecture, distributed bot fleet, sandboxing limits, telemetry collection, and the correctness verification engine.

What is QuanTime?

QuanTime is a high-performance distributed benchmarking and hosting platform for financial order books and matching engines. It automates compilation, sandbox deployment, real-time load testing, and telemetry collection to rank engine implementations on performance and correctness.

How is correctness verified?

Correctness is verified by an inline execution oracle that matches order submissions to a strict Price-Time Priority FIFO state representation. It audits fill accuracy, trade executions, order cancellations, and order state transition validity (no double-fills, accurate remaining quantity, correct pricing, valid sequence numbers).

What languages and runtimes are supported?

Any matching engine packaged with a Dockerfile can be benchmarked. We natively support and provide templates for Go, C++, and Rust matching engines, but any runtime that compiles inside a standard Linux container and exposes a REST/WebSocket API is compatible.

How does the sandboxing model work?

To ensure isolated and repeatable latency benchmarks, matching engines are run inside containerized sandboxes with restricted resource allocations: strict CPU pinning (via cgroups), limits on memory consumption, and no external internet access. Network communication is isolated on a dedicated bridge network.

What metrics does the telemetry ingester track?

Our telemetry system ingests high-frequency metrics including throughput (Transactions Per Second), p50/p90/p99 tail latencies, error/packet loss ratios, system resource utilization (CPU/Memory), and correctness failure events. Data is persisted to a TimescaleDB hypertable for real-time querying.

How does the bot fleet simulate market traffic?

The bot fleet is distributed across multiple load-generation instances communicating over a NATS JetStream event bus. They generate realistic market activity profiles-ranging from simple random walks to structured liquidity generation, cancellations, and heavy order bursts-to stress-test matching engine throughput bounds.

What does the AI Code Review analyzer check?

Our multi-agent LLM analysis system runs three specialized agents in parallel: a Security Agent auditing for container escape and network exploits, a Performance Agent identifying memory allocations and lock contention, and an Architecture Agent mapping design decisions. They synthesize a combined risk rating and actionable feedback.
